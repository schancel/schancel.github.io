Blog Post

Mining Without the Mempool

Draft — February 24, 2026

Seven years ago I wrote three essays describing the performance problem at the heart of Bitcoin’s block construction code, and submitted a patch to fix it. The patch was correct. It was benchmarked. It reduced codebase size. It was never merged.

This post is the postmortem. It also announces that the patch has been applied to Lotus — as part of a larger plan to remove the 25-transaction chain limit entirely.

The Problem

Bitcoin inherited a feature called Child Pays For Parent (CPFP). When a low-fee transaction is stuck in the mempool, a wallet can create a child transaction with a high fee, causing miners to include both together as a package. The pair earns more total fee than either alone.

Implementing this requires knowing the “package fee rate” of every transaction — the combined fees and combined size of the transaction plus all its unconfirmed ancestors. The mempool maintains this accounting eagerly: when a new transaction enters, it walks the entire ancestor chain and updates every entry. When a block arrives and confirmed transactions are removed, the accounting is updated again for all descendants.

For a chain of N transactions, each addition touches O(N) entries, so building a chain takes O(N²) total work. Processing a block that confirms N chained transactions takes O(N²) work. These are not theoretical worst cases — they are the normal case whenever wallets chain transactions.

Because O(N²) is a DoS vector, a hard limit was added: no chain longer than 25 transactions is admitted to the mempool. This protects nodes but also caps what applications can do with zero-confirmation transactions.

What Was Proposed in 2019

The insight: mining code does not need the mempool’s pre-computed CPFP fields. It can compute package statistics itself, at block-construction time, fast enough that the pre-computation becomes unnecessary.

The algorithm (D2866, submitted to Phabricator in 2019):

  1. Snapshot the mempool in insertion (topological) order. Parents always precede children in insertion order — this is maintained as an invariant. O(n)

  2. Forward walk the snapshot to accumulate package fee, size, and sigops for each transaction. One pass suffices because parents appear before children. O(n + m) where m is the number of parent-child edges.

  3. Assign each transaction a scheduling depth (called order). A transaction whose package fee rate exceeds that of any ancestor gets order = 0 — it goes first. Otherwise order is one deeper than its deepest ancestor. Sort ascending by order, then descending by fee rate. O(n log n)

  4. Visit in sorted order. For each transaction, walk its in-mempool ancestors via DFS and add them to the block before the transaction itself. O(n + m)

Total: O(n log n). The old algorithm was O(n²) from the progressive ancestor-score update loop inside addPackageTxs.

The benchmarks from 2019 were unambiguous. For a chain of 200 transactions, the new code was roughly 50× faster. For a chain of 50 transactions, roughly 10×. The improvement compounds with chain length — exactly the expected profile for eliminating a quadratic factor.

For blocks with no chained transactions at all (the common steady-state case), performance was indistinguishable.

Why It Wasn’t Merged

The patch went into review on Phabricator (D2866). It didn’t move.

I don’t have a complete explanation. The Bitcoin ABC team was under sustained political pressure during this period — the BCH/BSV split had just happened, there were ongoing disputes about the IFP and development funding, and the patch required touching both mempool and mining code, which makes reviewers cautious. The patch changed the observable semantics of package selection: under the old algorithm, a high-fee sibling could dynamically elevate a low-fee sibling’s package rate and pull it into a block. The new algorithm computes package stats upfront and doesn’t update them, so that sibling rescue doesn’t happen. The test suite had to be updated to reflect this.

That behavior change was probably the nail in the coffin. Reviewers who didn’t read the full analysis would see “test expectation inverted” and worry that mining revenue was being harmed. In practice the difference is negligible — the edge case requires an already-below-minimum-fee transaction to be rescued by a sibling, which is a niche scenario — but it was enough to stall review.

The methodology I proposed was a three-step plan:

  1. Make mining self-sufficient — block construction computes its own package stats. (D2866)
  2. Remove mempool ancestor/descendant bookkeeping. No pre-computation = no update loops = chain limit can be lifted.
  3. Clean up: remove the code, simplify eviction policy, delete tests.

Step 1 was the clean prerequisite. Steps 2 and 3 couldn’t happen without it, because the eviction logic (which drops transactions when the mempool is full) also depends on the package accounting. You need a replacement eviction policy before you can delete the accounting.

None of it landed.

What eCash and Bitcoin Cash Did Instead

Both eCash and Bitcoin Cash eventually raised or removed the 25-transaction chain limit. This is good. But neither fixed the underlying O(n²) construction.

Removing the limit without fixing the algorithm means: the DoS resistance that justified the limit is gone, but the quadratic work is still happening. A mempool with a 10,000-transaction chain does ten million times as much work per-block-construction as a chain of length 1. On most networks this doesn’t matter because deep chains rarely appear in practice. But it’s an unresolved technical debt, and any application that deliberately builds long chains (as CashWeb does) will eventually hit the ceiling — either the soft ceiling of performance degradation, or a reinstated hard limit when the problem becomes visible.

Why It Matters for Lotus

CashWeb is a messaging protocol built on Bitcoin transactions. It chains many transactions together as part of normal operation — each message or relay creates a new transaction spending the output of the previous. The 25-transaction limit is a hard constraint on how many hops a message can take without a confirmation. Removing that limit, and doing so correctly (without the O(n²) accumulation), is prerequisite to CashWeb working well at scale.

More generally, Lotus is designed for high-frequency zero-confirmation use. The Mitra transaction format, stateful script, and the Lotus denominations are all oriented toward applications that generate many transactions quickly. The mempool’s chain limit is architecturally incompatible with that goal.

Patch 29: What Was Applied

The patch applied to Lotus (patch 29 in the v2 series) is a direct port of D2866 to the current eCash codebase. Changes:

txmempool.h: Added a sequenced<insertion_order> index to CTxMemPool::indexed_transaction_set. This is a boost::multi_index sequenced index that tracks insertion order. It’s the data structure that makes step 1 of the algorithm possible — iterating the mempool in topological order.

miner.h: Extended CBlockTemplateEntry with package tracking fields (packageSize, packageFees, packageSigOps, maxPackageFeeRate, order) and methods (FeeRate(), AccountForParent()). The original 3-argument constructor is preserved for backward compatibility. A new 5-argument internal constructor is used during block assembly.

miner.cpp: New addPackageTxs (O(n log n)). The old function is renamed accuratelyAddPackageTxs and kept for comparison — not called in production. The new function implements the four-step algorithm above.

miner_tests.cpp: The package selection test that expected a low-fee sibling to be rescued by a high-fee sibling is inverted. Under the new algorithm, packages stats are fixed at snapshot time; the sibling rescue doesn’t occur. The test now verifies the new behavior.

The patch does not remove any mempool accounting code — that is step 2, which depends on also replacing the eviction policy. The current eviction strategy walks descendants by fee rate, which requires the same ancestor/descendant accounting that step 2 would remove. A replacement policy (probably: evict by individual fee rate, simplest correct option) needs to be designed and tested before step 2 can be written.

The Plan Forward

  • Patch 29 (done): mining no longer reads CPFP fields from the mempool. Block construction is self-sufficient.
  • Patch 30 (todo): remove MEMPOOL_ANCESTOR_LIMIT and MEMPOOL_DESCENDANT_LIMIT. This requires a replacement eviction policy that doesn’t depend on the package accounting being removed. Raising or eliminating the chain limit follows from this.
  • Patch 31 (todo): delete UpdateAncestorsOf, UpdateEntryForAncestors, CalculateDescendants, and the rest of the O(n²) machinery from CTxMemPool. Large net code deletion.

After patch 31, the mempool has a single responsibility: don’t accept orphan transactions. The package accounting burden is gone entirely.

On The Original Review

The patch was submitted in mid-2019 with full documentation, benchmarks, and a clear rationale. It sat unreviewed. This is not unusual for infrastructure work — reviewers who understand the code well enough to evaluate the change are the same people doing other development. The technical debt calcifies.

I don’t write this to relitigate old decisions. The Bitcoin ABC team was navigating a genuinely difficult period and I don’t know what their internal priorities were. But it is worth noting: the patch was sound. The three-step methodology was sound. The performance argument was backed by data. The only substantive change in behavior (the sibling rescue) was a documented tradeoff, not a regression. The code is smaller and faster.

The question of why technically correct, well-documented patches fail to get reviewed is worth thinking about. It isn’t unique to this case. Infrastructure work that reduces complexity rather than adding features tends to be undervalued in review queues. Code deletion is harder to celebrate than code addition. Performance improvements that don’t affect the common case get deprioritized. These are systematic biases in how open-source projects allocate review attention.

For Lotus, the answer is: we just apply the patch. We are not blocked by anyone else’s review queue.
